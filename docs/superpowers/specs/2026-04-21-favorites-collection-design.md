# Коллекция любимчиков — дизайн

Дата: 2026-04-21
Источник требований: Notion — «Коллекция любимчиков» (34989198-d9fd-80bb-9516-d542efb1f16e)

## Цель

Дать пользователю возможность сохранять настроенных мобов (основной + все пассажиры со всеми полями) и возвращаться к ним позже: загружать в редактор, копировать команду, переименовывать, удалять.

## Область

- Входит: сохранение/загрузка/переименование/удаление, копирование команды из любимчика.
- Не входит: экспорт/импорт JSON, синхронизация между устройствами, теги/категории, поиск.
- Координаты (`~ ~ ~` и т.п.) **не** сохраняются — они ситуативные.

## Хранилище

- `localStorage`, ключ `mc-summon:favorites`.
- Формат:
  ```js
  {
    version: 1,
    items: [
      {
        id: "fav-<timestamp>-<rand>",
        name: "Зомби-танк",
        createdAt: 1714000000000,
        mobOrder: [{ mobType: "zombie" }, { mobType: "skeleton" }],
        fields: { "0-mob": "zombie", "0-name": "Босс", "0-health": "200", "1-mob": "skeleton", ... }
      }
    ]
  }
  ```
- `fields` — плоский снапшот всех `data-field` элементов в `#mobs-container`, ключи хранятся с индексом моба (`0-name`, `1-health`, `0-slot-head`), **без** динамических id (см. «Пересчёт префиксов»).
- `mobOrder` — массив по индексам: 0 — основной, дальше пассажиры. Нужен для пересоздания карточек через существующий `createMobState` + `buildMobCard`.

### Ошибки хранилища

- `QuotaExceededError` при `setItem` → `alert('Не удалось сохранить: хранилище переполнено.')`, операция отменяется.
- Невалидный JSON при чтении → `console.warn`, возвращаем пустой список; пользователю показываем пустое состояние.
- `items[i].version !== 1` при загрузке конкретного любимчика → `alert('Этот любимчик сохранён в другой версии и не может быть загружен.')`, элемент остаётся в списке нетронутым.

## UI

Новая панель `⭐ Коллекция любимчиков` в `index.html`, **между** панелями «Координаты» и «Мобы».

### Заголовок
- `<h2>⭐ Коллекция любимчиков</h2>` + кнопка справа `⭐ Сохранить текущего`.

### Список
- Flex-сетка карточек (`.fav-card`) в стиле проекта — тёмный фон `var(--panel-2)`, рамка `var(--border)`, акцент `var(--accent-2)`.
- Карточка содержит:
  - Строка 1 (жирно): название любимчика.
  - Строка 2 (мелко, `var(--muted)`): `<русское имя основного моба> · +N пассажиров` (N опускается при N=0).
  - Строка 3 — ряд кнопок: `▶ Загрузить`, `✏`, `📋`, `🗑`.
- Пустое состояние: абзац с текстом «Пока пусто. Настрой моба и нажми ⭐ Сохранить текущего».

### Тост
- Маленький всплывающий блок внизу экрана для подтверждения «✓ Сохранено», «✓ Скопировано», «✓ Удалено». Авто-скрытие через 1.5с. Один общий `#toast` элемент.

## Поведение

### Сохранить
1. Клик `⭐ Сохранить текущего`.
2. `prompt('Название любимчика?', defaultName)`, где `defaultName` — русское название основного моба из `ALL_MOBS[mobs[0].mobType]`.
3. Пустая строка или отмена → ничего не делаем.
4. Иначе: `snapshotCurrent()` → добавить в `items` → `saveFavorites()` → `renderFavorites()` → тост «✓ Сохранено».
5. Дубликаты имён разрешены — не проверяем.

### Загрузить
1. Клик `▶ Загрузить` на карточке.
2. `confirm('Заменить текущую настройку на «<name>»?')` → если нет, выход.
3. `restoreSnapshot(item)`: пересоздать `mobs` по `mobOrder`, вызвать `renderAll()`, затем пройти по `fields` и восстановить значения с пересчётом префиксов.
4. Вызвать `generate()`. Координаты в `#coords` не трогать.

### Переименовать
1. Клик `✏`.
2. `prompt('Новое название?', item.name)`.
3. Пустая строка или отмена → выход. Иначе обновить, сохранить, перерисовать, тост «✓ Переименовано».

### Копировать команду
1. Клик `📋`.
2. `buildCommandFromSnapshot(item)` возвращает строку команды, собранную по тому же алгоритму, что и в `generate()`, с координатами `~ ~ ~`.
3. `navigator.clipboard.writeText(cmd)` → тост «✓ Скопировано».

### Удалить
1. Клик `🗑`.
2. `confirm('Удалить любимчика «<name>»?')` → при согласии: убрать из `items`, сохранить, перерисовать, тост «✓ Удалено».

## Функции (все добавляются в конец `<script>`)

- `loadFavorites() → {version, items}` — читает localStorage, ловит ошибки.
- `saveFavorites()` — пишет `{version:1, items}`, ловит `QuotaExceededError`.
- `snapshotCurrent() → {mobOrder, fields}` — собирает `fields` из всех `[data-field]` внутри `#mobs-container`, `mobOrder` из текущего `mobs[]`.
- `restoreSnapshot(item)` — пересоздаёт состояние.
- `buildCommandFromSnapshot(item) → string` — см. ниже.
- `renderFavorites()` — перерисовывает панель.
- `addFavorite()`, `loadFavorite(id)`, `renameFavorite(id)`, `copyFavoriteCommand(id)`, `removeFavorite(id)` — обработчики.
- `showToast(text)` — показывает `#toast`.

### Как собирать команду из снапшота без восстановления в DOM

Существующий `buildNBTPartsFor(prefix)` читает значения через `val(prefix, field)`, который работает с DOM. Минимальный рефакторинг:

- Ввести модульную переменную `fieldSource` (по умолчанию — DOM reader).
- `val()` делегирует `fieldSource(prefix, field)`.
- `buildCommandFromSnapshot` временно подменяет `fieldSource` на лямбду, читающую из `item.fields` (с пересчётом префиксов), собирает команду через `buildNBTPartsFor` по каждому мобу, возвращает строку, восстанавливает `fieldSource`.

Это оставляет один источник истины для построения NBT и не требует дублирования.

### Пересчёт префиксов

При сохранении `fields` содержит ключи вроде `m3-name`, `m3-health`, `m7-slot-head`. При загрузке новые карточки получат другие `id` (через `nextMobId++`). Решение:
- При сохранении: найти числовой `id` для каждого индекса в `mobOrder` (уже есть в текущем `mobs[]`), сохранять в ключах **индекс**, а не id: `fields["0-name"]`, `fields["1-health"]`.
- При восстановлении и чтении из снапшота: подменять префикс `m<id>-` на `<idx>-` обратно перед лукапом.

Это устраняет хрупкость и упрощает миграцию.

## Тестирование (ручное, Playwright MCP)

1. Сохранить текущего → карточка появилась в панели; перезагрузка страницы — карточка на месте.
2. Сбросить всё → загрузить любимчика → состояние и команда восстановлены полностью (проверить: имя, HP, слот, зачарование, пассажир, его отделка брони).
3. Переименовать — изменилось в UI и после перезагрузки.
4. Копировать команду — в буфере строка, идентичная той, что строит редактор на восстановленном состоянии (кроме координат).
5. Удалить — карточка исчезла, после перезагрузки не возвращается.
6. Пустое состояние — показывается корректно после удаления всех.

## Риски

- **Миграции формата**: `version: 1` сейчас, любое несовместимое изменение потребует миграции или сброса. Дизайн это учитывает.
- **Переполнение localStorage**: вряд ли в реалистичном сценарии (пара сотен любимчиков поместится), но ошибка обрабатывается.
- **Переименование полей в index.html**: если `data-field` имена изменятся, старые снапшоты частично потеряют данные. Принимаем как есть — не держим BC для внутреннего формата.
