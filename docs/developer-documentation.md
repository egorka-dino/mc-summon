# Техническая документация mc-commands

Документация для разработчиков и администраторов приложения `mc-commands`.

## Стек и окружение

- Сайт работает как Next.js App Router приложение на React и TypeScript.
- Деплой выполняется на Vercel.
- Авторизация подключена через Clerk (`@clerk/nextjs`).
- База данных для библиотек, серверных публикаций и служебных настроек подключается через Neon serverless (`@neondatabase/serverless`).
- Админский дашборд Exaroton читает серверы через публичный Exaroton API по `EXAROTON_API_KEY`.
- Общие стили публичных страниц и редакторов лежат в `style.css`.
- Данные и генерация команд разнесены по компонентам в `app/components/summon/` и `app/components/give/`.
- Favicon подключается через специальный файл App Router `app/icon.svg`.
- Production URL: `https://mc-commands.vercel.app`.
- Кастомный домен: `https://mc-commands.egorka.fun`.

## Основные маршруты приложения

- `/` — React-лендинг.
- `/servers` — публичная страница «Серверная» со списком опубликованных Minecraft-серверов Exaroton.
- `/locations` — публичный раздел «Места» для личных координат Minecraft.
- `/summon` — React-редактор команды `/summon`.
- `/give` — React-редактор команды `/give`.
- `/sign-in`, `/sign-up`, `/sign-out` — страницы авторизации Clerk.
- `/admin` — входное меню админки для пользователей с ролью `admin`.
- `/admin/servers` — админский дашборд серверов Minecraft на Exaroton.
- `/admin/library/items` — админский редактор библиотеки предметов для будущих публичных сценариев `/give`.
- `/admin/library/mobs` — админский редактор библиотеки готовых мобов для `/summon`.

## Генерация команд

### Общие редакторы

- `/give` использует клиентский компонент `app/components/give/ItemEditor.tsx`.
- `/admin/library/items` использует тот же `ItemEditor` в `adminMode`: поля команды «Кому выдать» и «Количество» скрыты, публичная коллекция из `localStorage` и блок выполнения на сервере скрыты, а preview и готовая команда остаются видимыми. Снимок предмета для библиотеки нормализуется к `target: "@s"` и `count: "1"`, чтобы количество задавалось уже в `/give` или в будущих сценариях.
- `/summon` использует клиентский компонент `app/components/summon/MobEditor.tsx`.
- `/admin/library/mobs` использует тот же `MobEditor` в `adminMode`: публичная коллекция любимчиков скрыта, а snapshot моба передаётся в форму библиотеки через `onSnapshotChange`.
- AI-помощник в публичных редакторах остаётся включённым. В админке библиотек он включается через `showAiAssistant`.

### `/summon`

- Команда собирается в `app/components/summon/engine.ts`.
- Состояние редактора хранится как `SummonSnapshot`: `mobOrder` плюс плоский словарь `fields` с ключами вида `<index>-<field>`.
- Снаряжение мобов собирается как единый блок `equipment:{head, chest, legs, feet, mainhand, offhand}`.
- Шансы выпадения снаряжения собираются в `drop_chances`.
- Зачарования предметов в снаряжении собираются в формате `components:{enchantments:{id:lvl}}`.
- Пассажиры собираются вложенным `Passengers:[...]`, где следующий моб сидит на предыдущем.
- Для фаерболлов добавляются `Power` и `ExplosionPower`.
- Тег босса добавляется как `Tags:["boss"]`.

### `/give`

- Команда собирается в `app/components/give/engine.ts`.
- Состояние редактора хранится как `GiveSnapshot`: выбранный предмет, цель, количество, `fields`, взрывы фейерверка, слои щита/флага и параметры зелья.
- Компоненты предметов собираются в квадратных скобках: `[comp=val,...]`.
- Зачарования собираются как `enchantments={id:lvl}`.
- Для Minecraft 1.21.5+ не используется старый формат `levels` внутри enchantments.
- Если итоговая команда длиннее 256 символов, цель `@s` автоматически заменяется на `@p`, но только когда пользователь не ввёл произвольную цель.
- Зелья представлены внутренними ID вида `potion:<effect>`, а итоговый item ID берётся из `potionType`: `potion`, `splash_potion`, `lingering_potion` или `tipped_arrow`.
- Компонент зелья собирается как `potion_contents={potion:"minecraft:<effect>"}`.

## Клиентские хранилища

- Коллекция `/summon` хранится в `localStorage` с ключом `mc-summon:favorites`.
- Коллекция `/give` хранится в `localStorage` с ключом `mc-give:favorites`.
- Анонимные места `/locations` хранятся в `localStorage` с ключом `mc-locations:places`.
- Эти коллекции локальны для браузера пользователя и не синхронизируются между устройствами.

## Авторизация

- Middleware находится в `proxy.ts`.
- Если Clerk-ключи настроены, общий `ClerkProvider` подключается в `app/admin/layout.tsx` для админских страниц. Это держит клиентскую Clerk-сессию активной в уже открытых вкладках админки и уменьшает риск устаревших cookies при долгой работе без полной перезагрузки страницы.
- Если Clerk-ключи не настроены, общий провайдер не монтируется, middleware пропускает запросы, а админка, `/sign-in`, `/sign-up` и `/auth/complete` показывают сообщение о необходимости настройки Clerk.
- Clerk middleware применяется только к auth/admin-поверхности: `/sign-in`, `/sign-up`, `/sign-out`, `/auth/complete`, `/admin`, `/admin/*`, `/api/admin/*` и `/api/auth/status`.
- Публичные страницы `/`, `/servers`, `/summon` и `/give` не вызывают `currentUser()` на сервере ради шапки и не проходят через Clerk middleware. Это сохраняет их HTML доступным без Clerk handshake для поисковиков, ChatGPT/browser-агентов и обычных анонимных пользователей.
- Угол авторизации на публичных страницах — клиентский компонент. После гидратации он запрашивает `/api/auth/status` с cookies текущего браузера и показывает имя, выход и ссылку «Админка» для пользователей с ролью `admin`.
- Пока статус авторизации загружается, угол авторизации держит пустой резерв места и не показывает кнопку входа, чтобы авторизованный пользователь не видел короткий неверный state.
- Страницы `/sign-in` и `/sign-up` читают только относительный `redirect_url`, отправляют Clerk на `/auth/complete?redirect_url=...` и используют `oauthFlow="redirect"`. Страница `/auth/complete` обёрнута в `ClerkProvider`, дожидается загрузки Clerk-сессии в браузере и только потом возвращает пользователя на исходную публичную страницу.
- Старый маршрут `/api/auth/google` перенаправляет на Clerk sign-in.
- Статус настройки Clerk и текущего пользователя отдаёт `/api/auth/status`.
- Админские права определяются по metadata пользователя в Clerk: `role: "admin"`, `admin: true` или наличие `admin` в `roles`.
- Приложение не доверяет `unsafeMetadata`, потому что эти данные может менять сам пользователь.

## Навигация

- Общая верхняя панель подключается в `app/layout.tsx` через `app/components/SiteNav.tsx`.
- Панель содержит ссылки на `/`, `/summon`, `/give`, `/locations` («Места») и `/servers` («Серверная»), подсвечивает активный раздел через `usePathname()` и показывает клиентский `AuthCorner`.
- Ссылка на `/admin` не входит в общий список навигации и остаётся только в `AuthCorner` для пользователей с ролью `admin`.
- Публичные страницы не дублируют локальные ссылки «назад»: возврат между разделами выполняется через верхнюю панель.

## Сборка

- `next.config.ts` явно задаёт `turbopack.root = process.cwd()`, чтобы Next/Vercel не пытались выводить корень проекта по lockfile в worktree-окружениях.

## Админка `/admin`

- Доступна только авторизованным пользователям с ролью `admin`.
- `/admin` показывает только меню разделов без выбранной страницы по умолчанию.
- Внутри админки есть отдельное меню разделов: `/admin/servers` для Exaroton, `/admin/library/items` для библиотеки предметов, `/admin/library/mobs` для библиотеки мобов и `/admin/scenarios` для сценариев.
- Не показывает служебные карточки текущего пользователя, healthcheck-проверок и публичных страниц.
- Позволяет управлять библиотекой предметов при настроенном `DATABASE_URL`.
- Для редактирования предмета использует тот же React-редактор, что и публичная страница `/give`, но в `adminMode` без публичной коллекции из `localStorage` и без блока выполнения команды на сервере. Предпросмотр и готовая команда остаются видимыми.
- В админском редакторе библиотеки предметов можно включать AI-помощник через `showAiAssistant`.
- Для выбранного предмета библиотеки можно выполнить `/give`: администратор выбирает online-сервер Exaroton, игрока и количество. Команда пересобирается из сохранённого snapshot на сервере.
- Позволяет управлять библиотекой мобов при настроенном `DATABASE_URL`.
- Для редактирования моба использует тот же React-редактор, что и публичная страница `/summon`, но в `adminMode` без публичного вывода команды и коллекции любимчиков.
- В админском редакторе библиотеки мобов можно включать AI-помощник через `showAiAssistant`.
- Для выбранного моба библиотеки можно выполнить `/summon`: администратор выбирает online-сервер Exaroton и место спавна — координаты, рядом с игроком или на позиции игрока.
- Позволяет управлять сценариями при настроенном `DATABASE_URL`. Сценарий состоит из действий: выдать предмет из библиотеки, призвать моба из библиотеки, выполнить другой сценарий. Структура действий оставляет место для будущих типов.

Возможности управления библиотекой мобов:

- Создать нового моба без ручного ввода ID.
- Выбрать существующего моба.
- Изменить категорию, название, описание, версию, включённость и snapshot `/summon`.
- Сохранить моба в Neon через `/api/admin/library/mobs`; для нового моба ID генерируется на сервере.
- Дублировать моба в новый черновик без ID.
- Удалить моба из библиотеки.
- Обновить список мобов вручную с сохранением текущего выбора, если он остался в базе.
- Предупреждать о несохранённых правках при уходе со страницы или выборе другого моба.

Возможности управления библиотекой предметов:

- Создать новый предмет без ручного ввода ID.
- Выбрать существующий предмет.
- Изменить категорию, название, описание, версию, включённость и snapshot `/give`.
- Сохранить предмет в Neon через `/api/admin/library/items`; для нового предмета ID генерируется на сервере.
- Дублировать предмет в новый черновик без ID.
- Удалить предмет из библиотеки.
- Обновить список предметов вручную с сохранением текущего выбора, если он остался в базе.
- Предупреждать о несохранённых правках при уходе со страницы или выборе другого предмета.

Возможности управления сценариями:

- Создать новый сценарий без ручного ввода ID.
- Выбрать существующий сценарий.
- Изменить название, описание, включённость и список действий.
- Добавить действие «Выдать предмет» из `library_items` с количеством `1-64`.
- Добавить действие «Призвать моба» из `library_mobs` с количеством `1-64` и местом спавна: рядом с игроком, на позиции игрока или координаты.
- Добавить действие «Выполнить сценарий» для вложенного запуска другого сценария.
- Дублировать, удалить и переставлять действия.
- Выполнить сценарий на выбранном Exaroton-сервере и увидеть результат по каждому действию.
- Предупреждать о несохранённых правках при уходе со страницы или выборе другого сценария.

## Библиотека мобов `/summon`

- Если настроен `DATABASE_URL`, `/summon` читает включённых готовых мобов из таблицы `library_mobs`.
- Админская страница `/admin/library/mobs` создаёт эту таблицу при первом обращении и сохраняет записи через общий серверный модуль `app/server/library-mobs.ts`.
- В публичный список попадают только записи с `enabled = true`; админская страница видит и скрытые записи.
- Публичный маршрут `/api/summon/templates` сохраняется как совместимый endpoint для текущего клиента, но отдаёт данные из библиотеки мобов. При проблемах с БД он возвращает пустой список, чтобы `/summon` не падал для пользователей.

## Библиотека предметов `/admin/library/items`

- Если настроен `DATABASE_URL`, админка хранит готовые предметы в таблице `library_items`.
- Таблица содержит `id`, `category`, `name`, `description`, `version`, `snapshot`, `enabled`, `created_at` и `updated_at`.
- `snapshot` хранит полный `GiveSnapshot`, поэтому команда всегда пересобирается через общий `buildGiveCommand`.
- Админская страница создаёт таблицу при первом обращении и сохраняет записи через `app/server/library-items.ts`.
- ID не редактируется в интерфейсе. Для новых предметов сервер генерирует ID через `createLibraryId("item", name)`.

## Библиотека мобов `/admin/library/mobs`

- Админка хранит готовых мобов в таблице `library_mobs`.
- Таблица содержит `id`, `category`, `name`, `description`, `version`, `mob_order`, `fields`, `enabled`, `created_at` и `updated_at`.
- `mob_order` и `fields` соответствуют `SummonSnapshot`, поэтому команда всегда пересобирается через общий `buildSummonCommand`.
- ID не редактируется в интерфейсе. Для новых мобов сервер генерирует ID через `createLibraryId("mob", name)`.
- Старое серверное хранилище `summon_mob_templates` и админский API `/api/admin/summon/templates` удалены. Единственный источник готовых мобов для админки и публичного `/summon` — `library_mobs`.

## Места `/locations`

- Публичная страница находится в `app/locations/page.tsx`.
- Клиентский редактор находится в `app/components/locations/LocationsEditor.tsx`.
- Общие типы, русские подписи, нормализация и валидация находятся в `app/components/locations/data.ts`.
- Анонимные места хранятся в `localStorage` с ключом `mc-locations:places`.
- Авторизованные места хранятся в Neon в таблице `player_locations` и всегда фильтруются по текущему Clerk user ID.
- Серверный модуль `app/server/locations.ts` создаёт таблицу при первом обращении.
- Клиент не передаёт и не выбирает `userId`; API берёт пользователя через `getAuthUser()`.
- Импорт локальных мест создаёт новые ID записей аккаунта, чтобы локальные ID не конфликтовали с серверными.
- Для выбора сервера редактор читает опубликованные серверы через `GET /api/servers/public`; если список недоступен или пользователь выбирает «Другой сервер», используется ручной ввод.

## Сценарии `/admin/scenarios`

- Админка хранит сценарии в таблице `scenarios`.
- Таблица содержит `id`, `name`, `description`, `enabled`, `actions`, `created_at` и `updated_at`.
- `actions` — JSON-массив действий. Поддерживаются `give_item`, `equip_player`, `summon_mob`, `run_scenario` и зарезервированный тип `future`.
- Сценарии не копируют формы `/give` и `/summon`: действия выбирают готовые элементы из `library_items` и `library_mobs`.
- Серверный executor находится в `app/server/scenario-execution.ts`.
- `equip_player` использует предмет из `library_items` и строит команду `item replace entity <player> <slot> with <item> 1`; поддерживаются слоты шлема, нагрудника, поножей, ботинок, основной и левой руки.
- В редакторе сценариев есть быстрые заготовки «Набор предметов», «Отряд мобов» и «Комплект игрока». Они создают обычные действия текущей модели и не вводят отдельное хранилище сценарных шаблонов.
- AI-заполнение сценариев доступно только в админке через `/api/admin/scenarios/ai`. AI возвращает черновик сценария из элементов библиотеки, не сохраняет его и не запускает команды.
- Если AI не находит подходящий предмет или моба в библиотеке, действие не добавляется, а админка показывает заметку с предложением создать элемент библиотеки.
- Проверка циклов вложенных сценариев вынесена в `app/shared/scenario-cycles.ts`, используется клиентским редактором и серверным сохранением.
- В UI нельзя выбрать вложенный сценарий, который замкнёт цикл; API сохранения дополнительно отклоняет такие сценарии с русским сообщением об ошибке.
- Вложенные сценарии разворачиваются перед выполнением. Циклы запрещены, максимальная глубина вложенности — 5.
- Выполнение идёт последовательно. При первой ошибке выполнение останавливается, а API возвращает результаты уже выполненных действий и ошибку текущего действия.

## Интеграция Exaroton

- Серверный клиент находится в `app/server/exaroton.ts`.
- Безопасная сборка команд выполнения находится в `app/server/minecraft-command-execution.ts`.
- Страница дашборда находится в `app/admin/servers/page.tsx`, а общая разметка карточек серверов — в `app/admin/exaroton-servers-panel.tsx`.
- Публичные генераторы `/give` и `/summon`, а также админские страницы библиотеки подключают общий клиентский блок `app/components/ServerCommandExecutor.tsx`. Он виден только администраторам, потому что читает защищённый маршрут `/api/admin/minecraft/servers`; для `401/403` блок не отрисовывается.
- Для авторизации используется одна переменная окружения: `EXAROTON_API_KEY`.
- Админка вызывает `GET https://api.exaroton.com/v1/servers/` на сервере, поэтому API-ключ не попадает в клиентский JavaScript.
- Выполнение сгенерированных команд использует `POST https://api.exaroton.com/v1/servers/{server}/command/` на сервере, поэтому API-ключ не попадает в клиентский JavaScript.
- Страница `/admin/servers` помечена как `dynamic = "force-dynamic"`, а запрос к Exaroton выполняется с `cache: "no-store"`, чтобы при открытии показывать актуальные статусы и игроков.
- Настройки публикации серверов хранятся в Neon в таблице `exaroton_server_publications` с ключом `server_id` и флагом `published`.
- В `/admin/servers` администратор может отметить сервер как опубликованный. Страница `/servers` показывает только опубликованные серверы из живого списка Exaroton.
- Если ключ не настроен, API недоступен, сервер offline или список игроков не отдаётся Exaroton API, админка и блок выполнения показывают понятное состояние вместо падения страницы. В блоке выполнения отображаются только серверы со статусом online.
- `ServerCommandExecutor` всегда показывает кнопку «Обновить серверы» после первой загрузки, включая состояния ошибки и пустого списка online-серверов.
- API выполнения не принимает произвольную команду от клиента. Он принимает только `mode`, `serverId`, `player`, snapshot текущего генератора или библиотеки, опциональное `count` для `/give` и опциональный `summonSpawn` для `/summon`, заново строит команду через `buildMinecraftExecutionCommand`, проверяет online-статус сервера и при необходимости наличие игрока в списке Exaroton, а затем отправляет в Exaroton команду без начального `/`.
- Для `/give` API пересобирает команду с выбранным игроком как целью и количеством `1-64`. Для `/summon` API поддерживает `execute at <player> run summon ...` для спавна у игрока и прямой `summon ... x y z` для координатного спавна без выбора игрока.

## API

- `GET /api/health` — healthcheck Next.js runtime.
- `GET /api/db/health` — healthcheck подключения к Neon Postgres.
- `GET /api/auth/status` — статус настройки Clerk и текущего пользователя.
- `GET /api/auth/google` — совместимый редирект старого Google-входа на Clerk sign-in.
- `GET /api/summon/templates` — публичный список включённых готовых мобов для `/summon`.
- `GET /api/locations` — список мест текущего пользователя.
- `POST /api/locations` — создание или обновление места текущего пользователя.
- `DELETE /api/locations?id=...` — удаление места текущего пользователя.
- `POST /api/locations/import` — импорт локальных мест в аккаунт.
- `GET /api/servers/public` — публичный список опубликованных серверов для выбора в пользовательских формах.
- `GET /api/admin/servers/publication` — админский список настроек публикации серверов Exaroton.
- `POST /api/admin/servers/publication` — админское переключение публикации сервера Exaroton на главной странице.
- `GET /api/admin/library/items` — админский список предметов библиотеки.
- `POST /api/admin/library/items` — создание или обновление предмета библиотеки, доступно только админам.
- `DELETE /api/admin/library/items?id=...` — удаление предмета библиотеки, доступно только админам.
- `GET /api/admin/library/mobs` — админский список мобов библиотеки, включая скрытые записи.
- `POST /api/admin/library/mobs` — создание или обновление моба библиотеки, доступно только админам.
- `DELETE /api/admin/library/mobs?id=...` — удаление моба библиотеки, доступно только админам.
- `GET /api/admin/scenarios` — админский список сценариев вместе со справочниками предметов и мобов для редактора.
- `POST /api/admin/scenarios` — создание или обновление сценария, доступно только админам.
- `DELETE /api/admin/scenarios?id=...` — удаление сценария, доступно только админам.
- `POST /api/admin/scenarios/ai` — AI-заполнение черновика сценария из библиотеки, доступно только админам; не сохраняет и не выполняет сценарий.
- `POST /api/admin/scenarios/execute` — выполнение сценария на выбранном Exaroton-сервере с результатом по каждому действию.
- `GET /api/admin/minecraft/servers` — админский список серверов и онлайн-игроков Exaroton для блока «Выполнить на сервере».
- `POST /api/admin/minecraft/execute` — админское выполнение команды, пересобранной из snapshot генератора `/give` или `/summon`; произвольный текст команды не принимается.
- `POST /api/ai/command` — серверный AI-помощник для режимов `summon` и `give`.

## AI-помощник

- Реализован в `app/api/ai/command/route.ts`.
- Требует `OPENAI_API_KEY`.
- Модель по умолчанию: `gpt-4.1-mini`; можно переопределить через `OPENAI_MODEL`.
- Для Responses API передаёт стабильный `prompt_cache_key` отдельно для режимов `summon` и `give`, чтобы повторяющийся системный prompt, схема ответа и справочники Minecraft попадали в prompt cache.
- Опциональная переменная `OPENAI_PROMPT_CACHE_RETENTION` принимает `in_memory` или `24h`; если не задана, используется политика OpenAI по умолчанию.
- При отсутствии `OPENAI_API_KEY` возвращает понятную ошибку и не блокирует ручные редакторы.
- Поддерживает режимы `summon` и `give`.
- Для сценариев отдельный админский route `app/api/admin/scenarios/ai/route.ts` использует ту же переменную `OPENAI_API_KEY`, передаёт модели только списки доступных элементов библиотеки и валидирует результат через `app/server/scenario-ai.ts`.
- Ограничивает длину prompt до 1500 символов.
- Запрашивает у модели JSON по компактной схеме операций, а не готовую команду.
- В AI-схеме обязательна только `operations`; `summary`, `notes` и поля внутри operation опциональны. Модель должна не возвращать `null` и неиспользуемые поля, а сервер трактует отсутствующие значения через дефолты и валидацию.
- После ответа применяет только разрешённые ID и поля: мобов, предметов, эффектов, чар, слотов, цветов, отделок, узоров, типов зелий и целей.

## Локальная разработка

```bash
npm install
npm run dev
```

Для базы данных и авторизации скопировать `.env.example` в `.env.local` и заполнить значения:

```bash
cp .env.example .env.local
```

Локальные переменные из Vercel можно обновить командой:

```bash
npx vercel@latest env pull .env.local
```

Production build:

```bash
npm run build
```

Production start после build:

```bash
npm run start
```

Если проверка функциональности зависит от Clerk/Neon или локально auth/db мешают нормальной валидации, нужно делать Vercel preview deployment и проверять preview URL.
