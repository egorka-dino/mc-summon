# Техническая документация mc-commands

Документация для разработчиков и администраторов приложения `mc-commands`.

## Стек и окружение

- Сайт работает как Next.js App Router приложение на React и TypeScript.
- Деплой выполняется на Vercel.
- Авторизация подключена через Clerk (`@clerk/nextjs`).
- База данных для общих шаблонов подключается через Neon serverless (`@neondatabase/serverless`).
- Админский дашборд Exaroton читает серверы через публичный Exaroton API по `EXAROTON_API_KEY`.
- Общие стили публичных страниц и редакторов лежат в `style.css`.
- Данные и генерация команд разнесены по компонентам в `app/components/summon/` и `app/components/give/`.
- Production URL: `https://mc-commands.vercel.app`.
- Кастомный домен: `https://mc-commands.egorka.fun`.

## Основные маршруты приложения

- `/` — React-лендинг.
- `/summon` — React-редактор команды `/summon`.
- `/give` — React-редактор команды `/give`.
- `/sign-in`, `/sign-up`, `/sign-out` — страницы авторизации Clerk.
- `/admin` — админка для пользователей с ролью `admin`.
- `/admin/servers` — админский дашборд серверов Minecraft на Exaroton.

## Генерация команд

### `/summon`

- Команда собирается в `app/components/summon/engine.ts`.
- Состояние редактора хранится как `SummonSnapshot`: `mobOrder` плюс плоский словарь `fields` с ключами вида `<index>-<field>`.
- Снаряжение мобов собирается как единый блок `equipment:{head, chest, legs, feet, mainhand, offhand}`.
- Шансы выпадения снаряжения собираются в `drop_chances`.
- Зачарования предметов в снаряжении собираются в формате `components:{enchantments:{id:lvl}}`.
- Пассажиры собираются вложенным `Passengers:[...]`, где следующий моб сидит на предыдущем.
- Для фаерболлов добавляются `Power` и `ExplosionPower`.
- Тег босса добавляется как `Tags:["boss"]`.

### `/give`

- Команда собирается в `app/components/give/engine.ts`.
- Состояние редактора хранится как `GiveSnapshot`: выбранный предмет, цель, количество, `fields`, взрывы фейерверка, слои щита/флага и параметры зелья.
- Компоненты предметов собираются в квадратных скобках: `[comp=val,...]`.
- Зачарования собираются как `enchantments={id:lvl}`.
- Для Minecraft 1.21.5+ не используется старый формат `levels` внутри enchantments.
- Если итоговая команда длиннее 256 символов, цель `@s` автоматически заменяется на `@p`, но только когда пользователь не ввёл произвольную цель.
- Зелья представлены внутренними ID вида `potion:<effect>`, а итоговый item ID берётся из `potionType`: `potion`, `splash_potion`, `lingering_potion` или `tipped_arrow`.
- Компонент зелья собирается как `potion_contents={potion:"minecraft:<effect>"}`.

## Клиентские хранилища

- Коллекция `/summon` хранится в `localStorage` с ключом `mc-summon:favorites`.
- Коллекция `/give` хранится в `localStorage` с ключом `mc-give:favorites`.
- Эти коллекции локальны для браузера пользователя и не синхронизируются между устройствами.

## Авторизация

- Middleware находится в `proxy.ts`.
- Если Clerk-ключи не настроены, middleware пропускает запросы, а админка показывает сообщение о необходимости настройки Clerk.
- Старый маршрут `/api/auth/google` перенаправляет на Clerk sign-in.
- Статус настройки Clerk и текущего пользователя отдаёт `/api/auth/status`.
- Админские права определяются по metadata пользователя в Clerk: `role: "admin"`, `admin: true` или наличие `admin` в `roles`.
- Приложение не доверяет `unsafeMetadata`, потому что эти данные может менять сам пользователь.

## Админка `/admin`

- Доступна только авторизованным пользователям с ролью `admin`.
- Показывает данные текущего пользователя: имя, email и Clerk ID.
- Даёт быстрые ссылки на healthcheck-эндпоинты и публичные страницы.
- Даёт ссылку на отдельную страницу `/admin/servers` с дашбордом Exaroton.
- Позволяет управлять общими шаблонами мобов для `/summon` при настроенном `DATABASE_URL`.
- Для редактирования шаблона использует тот же React-редактор мобов, что и публичная страница `/summon`, но в `adminMode` без публичного вывода команды и коллекции любимчиков.
- В админском редакторе можно использовать AI-помощник «Собрать шаблон словами».

Возможности управления шаблонами:

- Создать новый шаблон.
- Выбрать существующий шаблон.
- Изменить ID, категорию, название, описание, версию, включённость и snapshot мобов.
- Сохранить шаблон в Neon через `/api/admin/summon/templates`.
- Скопировать шаблон в новый черновик.
- Удалить DB-запись шаблона; для встроенных шаблонов это возвращает fallback-вариант из кода.
- Предупреждать о несохранённых правках при уходе со страницы или выборе другого шаблона.

## Общие шаблоны `/summon`

- Встроенные шаблоны остаются в React-данных `/summon` как fallback.
- Если настроен `DATABASE_URL`, админка создаёт и использует таблицу `summon_mob_templates` в Neon.
- Запись с тем же `id`, что у встроенного шаблона, подменяет его для всех пользователей.
- Удаление DB-записи возвращает встроенный вариант.
- Публичный маршрут `/api/summon/templates` при проблемах с БД возвращает пустой список, чтобы `/summon` не падал для пользователей.

## Интеграция Exaroton

- Серверный клиент находится в `app/server/exaroton.ts`.
- Страница дашборда находится в `app/admin/servers/page.tsx`, а общая разметка карточек серверов — в `app/admin/exaroton-servers-panel.tsx`.
- Для авторизации используется переменная окружения `EXAROTON_API_KEY`; также поддерживаются имена `EXAROTON_API_TOKEN` и `EXAROTON_TOKEN`.
- Админка вызывает `GET https://api.exaroton.com/v1/servers/` на сервере, поэтому API-ключ не попадает в клиентский JavaScript.
- Страница `/admin/servers` помечена как `dynamic = "force-dynamic"`, а запрос к Exaroton выполняется с `cache: "no-store"`, чтобы при открытии показывать актуальные статусы и игроков.
- Если ключ не настроен, API недоступен или список игроков не отдаётся Exaroton API, админка показывает понятное состояние вместо падения страницы.

## API

- `GET /api/health` — healthcheck Next.js runtime.
- `GET /api/db/health` — healthcheck подключения к Neon Postgres.
- `GET /api/auth/status` — статус настройки Clerk и текущего пользователя.
- `GET /api/auth/google` — совместимый редирект старого Google-входа на Clerk sign-in.
- `GET /api/summon/templates` — публичный список включённых шаблонов мобов для `/summon`.
- `GET /api/admin/summon/templates` — админский список шаблонов, включая служебные данные.
- `POST /api/admin/summon/templates` — создание или обновление шаблона моба, доступно только админам.
- `DELETE /api/admin/summon/templates?id=...` — удаление DB-записи шаблона, доступно только админам.
- `POST /api/ai/command` — серверный AI-помощник для режимов `summon` и `give`.

## AI-помощник

- Реализован в `app/api/ai/command/route.ts`.
- Требует `OPENAI_API_KEY`.
- Модель по умолчанию: `gpt-4.1-mini`; можно переопределить через `OPENAI_MODEL`.
- Для Responses API передаёт стабильный `prompt_cache_key` отдельно для режимов `summon` и `give`, чтобы повторяющийся системный prompt, схема ответа и справочники Minecraft попадали в prompt cache.
- Опциональная переменная `OPENAI_PROMPT_CACHE_RETENTION` принимает `in_memory` или `24h`; если не задана, используется политика OpenAI по умолчанию.
- При отсутствии `OPENAI_API_KEY` возвращает понятную ошибку и не блокирует ручные редакторы.
- Поддерживает режимы `summon` и `give`.
- Ограничивает длину prompt до 1500 символов.
- Запрашивает у модели JSON по компактной схеме операций, а не готовую команду.
- В AI-схеме обязательна только `operations`; `summary`, `notes` и поля внутри operation опциональны. Модель должна не возвращать `null` и неиспользуемые поля, а сервер трактует отсутствующие значения через дефолты и валидацию.
- После ответа применяет только разрешённые ID и поля: мобов, предметов, эффектов, чар, слотов, цветов, отделок, узоров, типов зелий и целей.

## Локальная разработка

```bash
npm install
npm run dev
```

Для базы данных и авторизации скопировать `.env.example` в `.env.local` и заполнить значения:

```bash
cp .env.example .env.local
```

Локальные переменные из Vercel можно обновить командой:

```bash
npx vercel@latest env pull .env.local
```

Production build:

```bash
npm run build
```

Production start после build:

```bash
npm run start
```

Если проверка функциональности зависит от Clerk/Neon или локально auth/db мешают нормальной валидации, нужно делать Vercel preview deployment и проверять preview URL.

## Правило сопровождения

Если меняется техническое поведение приложения, API, админка, авторизация, БД, хранилища, интеграции или формат генерации команд, этот документ нужно обновить в той же задаче.

Если изменение также влияет на публичный пользовательский сценарий, нужно обновить `docs/user-functionality.md`.
